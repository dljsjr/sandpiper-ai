# Per-extension postinstall strategy for extensions with npm dependencies

Investigate unbundling extensions to align with Pi's happy-path extension loading.

## Background

Pi's extension loader uses jiti to load .ts files directly — no build step needed.
Extensions declare their entry point and jiti handles TypeScript compilation and
module resolution at runtime. This is the 'happy path' for Pi extensions.

Our extensions currently deviate from this in different ways:
- system.ts — loaded directly from source (happy path ✅)
- shell-relay — pre-bundled via 'bun build' into a single .js file
- web-fetch — compiled via tsc to dist/, loaded from dist/index.js

The pre-bundling approach creates confusion about how dependencies are resolved,
what ends up in the dist/ directory, and how the postinstall script works.

## The Question

Can we 'unbundle' shell-relay and web-fetch to load directly from source .ts files,
the same way system.ts does? This would:
- Align with Pi's happy-path extension loading
- Eliminate per-extension build steps
- Simplify postinstall (just copy source files to dist/)
- Make the dependency story clearer (jiti resolves from hoisted node_modules)

## Key Considerations

- Extensions with npm dependencies (web-fetch: jsdom, readability, turndown) need
  those deps available at runtime. With hoisted linker in bunfig.toml, they're in
  root node_modules/ — jiti can resolve them. BUT for non-dev installs (when
  sandpiper is distributed as a package), these deps may not be present.

- Shell-relay's companion scripts (ghost-attach, unbuffer-relay) are standalone
  shell scripts, NOT part of the JS bundle. They're copied to dist/ by postinstall.
  (Note: ghost-attach and unbuffer-relay are now deprecated — the Zellij 0.44
  rewrite eliminated them. But shell-integration/ scripts still need to be copied.)

- The dist/package.json generated by postinstall declares the Pi Package resource
  paths. It does NOT manage npm dependencies for extensions — that's handled by
  the workspace package.json files and bun install.

- jiti's module resolution in Pi's extension loader uses aliases for known packages
  (@mariozechner/pi-coding-agent, pi-tui, etc.) and standard node_modules resolution
  for everything else. Our tsconfig.json paths entries help TypeScript type-checking
  but are NOT used by jiti at runtime.
